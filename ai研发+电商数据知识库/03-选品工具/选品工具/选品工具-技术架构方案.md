## 二、前端技术选型

| 类别 | 选型 | 理由 |
|------|------|------|
| 框架 | Vue 3 + TypeScript | 已确定 |
| 构建工具 | Vite | 极快热更新，开发体验好 |
| UI 组件库 | Element Plus | 生态最成熟，文档完善 |
| 图表库 | ECharts + vue-echarts | 趋势图首选，中文友好 |
| 状态管理 | Pinia | Vue 3 官方推荐，简洁 |
| 请求库 | Axios | 拦截器，统一错误处理 |
| 路由 | Vue Router 4 | 动态路由，权限控制 |
| CSS 方案 | UnoCSS | 原子化CSS，轻量 |
| 工具库 | @vueuse/core | 实用组合式函数集合 |

**项目结构：**
```
src/
├── api/          # 接口定义
├── components/   # 公共组件
├── views/        # 页面视图
├── stores/       # Pinia store
├── composables/  # 组合式函数
├── types/        # TypeScript 类型
└── utils/        # 工具函数
```

## 三、后端技术选型

| 类别 | 选型 | 理由 |
|------|------|------|
| 语言/框架 | Node.js + NestJS | 与前端统一 JS 生态，小团队省力 |
| 数据库 | PostgreSQL | 复杂查询，支持 JSON 字段，够用不引入 MongoDB |
| 缓存 | Redis | API 结果缓存，防限流，分布式锁 |
| ORM | Prisma | 类型安全，AI 辅助开发友好 |
| 任务队列 | Bull（基于 Redis） | 定时采集 + 推送任务 |
| 定时任务 | @nestjs/schedule | Cron 任务管理 |
| 认证 | Passport.js + JWT | 标准方案 |
| API 文档 | Swagger（OpenAPI） | 自动生成，OpenClaw 接入参考 |
| 日志 | Pino | 结构化日志，性能好 |

## 四、数据采集层设计

| 平台 | 获取方式 | 频率 | 备注 |
|------|---------|------|------|
| Reddit | 官方 API (PRAW) | 每天 | 速率限制 60 req/min，需缓存 |
| Google Trends | pytrends 库 | 每天 | 高频触发限流，需随机间隔 |
| Facebook 广告库 | Ad Library API（官方免费） | 每小时 | 需 Meta 开发者账号 |
| TikTok | Apify 第三方服务（降级方案） | 每天 | Research API 审核周期长，先用第三方 |

### Reddit AI 翻译摘要流程

```
Reddit API 拉取原始帖子
    → 内容过滤（去低质量/无关内容）
    → AI 翻译 + 摘要（中文一句话结论）
    → 存入 PostgreSQL + Redis 缓存
    → 前端展示
```

### OpenClaw 专用 API 设计原则

> [!warning] 关键约束
> OpenClaw Agent 操作网页时不能依赖 DOM 结构，前端改版会导致 agent 失效。
> 必须提供独立稳定的 RESTful API 接口层。

```
GET  /api/v1/agent/trends          # 获取趋势数据
GET  /api/v1/agent/products        # 获取选品列表
POST /api/v1/agent/analyze         # 提交分析任务
GET  /api/v1/agent/results/:taskId # 获取分析结果
WS   /ws/agent/updates             # 实时推送
```

## 六、风险清单

### 🔴 高风险

| 风险 | 问题 | 应对措施 |
|------|------|---------|
| 第三方 API 不稳定 | TikTok/Reddit 随时失效 | 多源备份 + 降级策略 + 熔断机制 |
| AI API 成本失控 | 高并发下翻译/分析费用爆炸 | 严格限流 + 缓存 + 预算告警 |
| Agent API 被滥用 | 恶意爬取数据 | API Key 认证 + IP 白名单 + 频率限制 |

### 🟡 中风险

| 风险 | 问题 | 应对措施 |
|------|------|---------|
| 定时任务重复执行 | 多实例部署导致数据重复 | Redis 分布式锁 |
| 推送渠道失效 | 微信/飞书 Webhook 失效 | 多通道冗余 + 失败重试 |
| 时区问题 | Reddit/Google 数据为 UTC，显示错乱 | 统一 UTC 存储，前端按用户时区转换 |

### 🟢 改进点

| 风险 | 应对措施 |
|------|---------|
| 首屏加载慢 | Vite 打包优化 + 骨架屏 |
| 数据库连接池耗尽 | 合理配置连接数 + 监控告警 |
| 冷启动数据为空 | 预置示例数据兜底 |

## 七、关键决策说明

> [!note] Claude vs Codex 分歧与采纳理由
> - **后端 Node.js**（采纳 Codex）：全栈 JS 统一，小团队 AI 辅助开发更顺畅
> - **不引入 MongoDB**（采纳 Claude）：PostgreSQL JSON 字段完全够用，早期减少复杂度
> - **Element Plus**（双方共识）：生态最成熟
> - **Vercel + Railway 起步**（采纳 Codex）：零运维成本，适合 MVP 阶段
> - **时区统一处理**（双方盲区，Claude 补充）：UTC 存储，前端转换
> - **Redis 分布式锁**（双方盲区，Claude 补充）：防定时任务重复执行
