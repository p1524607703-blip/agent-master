# Phase 2：文案撰写系统 — 详细 PRD

> [!info] 上下文
> 本文档为 [[TikTok广告AI工作流-PRD总览-2026-03-29]] 的 Phase 2 子文档。
> **依赖**：Phase 1 创意拆解插件输出的结构化标签数据库。
> **产出**：供 Phase 3 出图使用的 `ImageGenerationBrief` JSON。

---

## 一、产品目标

**一句话定义**：把 Phase 1 积累的竞品创意数据库，自动提炼为高转化文案公式，批量生成候选文案，经文案策划筛选后驱动出图与视频合成。

**解决的核心痛点**：

| 痛点 | 当前状态 | 期望状态 |
|------|---------|---------|
| 文案依赖个人经验 | 文案策划凭感觉写，质量不稳定 | 从竞品数据库提炼高频公式，有数据支撑 |
| 生成速度慢 | 人工写一批文案需1-2小时 | Claude 批量生成20条，< 5分钟 |
| 无评审闭环 | 好文案不回流，差文案不淘汰 | 评审结果自动写回数据库，质量持续迭代 |
| 文案和出图脱节 | 文案写完给设计，设计自由发挥 | 文案直接输出结构化 Brief 驱动出图参数 |

---

## 二、用户与使用场景

**主要用户**：文案策划（每天需产出3-5套可用广告文案，目前人工完成）

**使用场景**：

| 场景 | 触发 | 产出 |
|------|------|------|
| 日常文案生产 | 打开文案系统 → 选择今日主推产品 → 一键生成 | 20条候选文案，选3-5条进入出图流程 |
| 新品上市 | 输入新产品卖点 → 生成文案 | 基于同类竞品数据库的新品文案 |
| 大促冲量 | 选择"大促模式" → 生成紧迫感文案 | 限时/折扣/库存稀缺类文案 |
| 复盘迭代 | 查看历史采用率 → 调整 Prompt 版本 | 文案生成质量持续提升 |

## 四、冷启动策略

> [!warning] 冷启动是系统上线第一周的最大风险
> Phase 2 的文案质量依赖 Phase 1 数据库的积累量。数据库为空时直接调用 Claude 会生成泛泛文案，和不用 AI 没区别。

### 4.1 冷启动阈值

| 数据库条目数 | 系统模式 | Claude Prompt 策略 |
|------------|---------|------------------|
| 0 - 49 条 | **冷启动模式**（兜底） | 使用行业通用公式模板，不检索数据库 |
| 50 - 199 条 | **成长模式** | 检索数据库高频公式 + 行业模板混合 |
| 200 条以上 | **正常模式** | 完全基于自有数据库，数据驱动 |

系统在每次生成前自动判断当前模式，顶部状态栏显示：
`📊 创意数据库：47条（冷启动模式，建议继续用插件积累）`

### 4.2 种子数据准备（上线前必做）

> [!success] 操作清单（Sprint 0，开发并行进行）
> - [ ] 由竞品分析员人工整理30-50条历史高转化竞品视频
> - [ ] 使用 Phase 1 插件对每条视频执行分析，写入数据库
> - [ ] 目标：系统上线当天数据库 ≥ 50条，直接进入成长模式

### 4.3 行业通用公式模板（冷启动兜底）

冷启动期间使用以下预置公式，绕过数据库检索：

```
鞋类广告高转化公式库（内置，不依赖数据库）：

公式1（痛点型）：[场景痛点] → [产品特性] → [结果情绪] → [CTA]
示例："站了8小时，同事都在换鞋，我还没感觉 → 记忆棉中底 → 脚不疼才是真的舒服 → 评论问尺码"

公式2（社交证明型）：[他人认可] → [产品亮点] → [用户收益] → [CTA]
示例："出门被问了3次在哪买的 → 独家设计款 → 好看又舒服 → 点购物车"

公式3（前后对比型）：[before状态] → [转折] → [after状态] → [CTA]
示例："换了这双之前走路真的很累 → 换了之后 → 走路像踩云，一整天不喊脚疼 → 限时价"

公式4（好奇悬念型）：[反直觉陈述] → [揭晓] → [产品呈现] → [CTA]
示例："为什么这双鞋卖了100万双 → 因为它真的很轻 → 才185g，比手机还轻 → 点击看实测"

公式5（大促紧迫型）：[限时信号] → [价值陈述] → [损失厌恶] → [强CTA]
示例："最后48小时 → 这个价格今年只有这一次 → 错过要等明年 → 立即抢购"
```

## 五、文案生成流程

### 5.1 输入参数

文案策划在生成前填写以下参数（UI 表单）：

| 参数 | 类型 | 示例 | 必填 |
|------|------|------|------|
| 产品名称 | 文本 | "PRIME X EVO 运动鞋" | ✅ |
| 核心卖点（1-3个） | 多选标签 | 轻量 / 舒适 / 外观 | ✅ |
| 目标人群 | 单选 | 上班族 / 运动爱好者 / 时尚人群 | ✅ |
| 情绪方向 | 单选 | 认同共鸣 / 好奇探索 / 社交展示 | ✅ |
| 文案模式 | 单选 | 日常种草 / 大促冲量 / 新品发布 | ✅ |
| 参考竞品公式 | 可选，从数据库选 | （下拉选历史高效公式） | ❌ |
| 生成数量 | 数字 | 20（默认） | ✅ |

### 5.2 数据库检索逻辑（成长/正常模式）

```python
# 从 Phase 1 数据库检索高频有效创意公式
def retrieve_creative_patterns(target_audience, emotion_direction, top_n=5):
    # 筛选条件：
    # 1. 同一目标人群
    # 2. 情绪弧线包含所选方向
    # 3. 历史采用率 > 0（曾被文案策划采用过）
    # 4. 按"采用率 × 点赞数权重"排序
    patterns = db.query(
        audience=target_audience,
        emotion__contains=emotion_direction,
        adopted_count__gt=0
    ).order_by('-score').limit(top_n)

    return [p.creative for p in patterns]
```

### 5.3 Claude API 调用

```javascript
// 文案生成核心调用
async function generateCopyBatch(params, patterns) {
  const patternContext = patterns.length > 0
    ? `以下是从竞品数据库中检索的高效创意公式（按转化率排序）：\n${JSON.stringify(patterns, null, 2)}`
    : `（当前使用行业通用公式模板，数据库积累中）`;

  const response = await claudeAPI.messages.create({
    model: 'claude-sonnet-4-6',   // 文案生成用 Sonnet（质量优先）
    max_tokens: 3000,
    system: COPY_SYSTEM_PROMPT,   // 见 5.4 节
    messages: [{
      role: 'user',
      content: `
产品信息：
- 产品名称：${params.productName}
- 核心卖点：${params.sellingPoints.join('、')}
- 目标人群：${params.audience}
- 情绪方向：${params.emotion}
- 文案模式：${params.mode}

${patternContext}

请生成 ${params.count} 条广告文案候选，每条包含：
钩子句、正文、CTA，并标注使用的创意公式类型。
输出严格的 JSON 数组格式。
      `
    }]
  });

  return JSON.parse(response.content[0].text);
}
```

### 5.4 文案生成系统 Prompt

```
你是专业的 TikTok 跨境鞋类广告文案专家，深度理解东南亚/欧美市场。

【核心原则】
1. 每条文案必须在前3秒内抓住注意力（钩子句）
2. 情绪驱动优先于功能描述
3. 口语化、接地气，避免品牌官方腔调
4. CTA 要具体可执行（不要写"点击了解"，要写"评论问尺码"/"点购物车看尺码"）
5. 总字数控制在80字以内（TikTok 字幕显示限制）

【输出格式】严格输出 JSON 数组，每条结构如下：
[
  {
    "id": 1,
    "formula_type": "痛点激发|社交证明|前后对比|好奇悬念|大促紧迫",
    "hook": "<前3秒钩子句，不超过20字>",
    "body": "<正文，展开产品核心卖点>",
    "cta": "<行动引导，具体动作>",
    "full_copy": "<完整文案，hook+body+cta连贯版本>",
    "emotion_target": "<这条文案狙击的情绪>",
    "estimated_audience_fit": "<最匹配的人群描述>"
  }
]

不要输出任何 JSON 以外的内容。
```

> [!tip] 模型选择说明
> 文案生成使用 `claude-sonnet-4-6`（非 Haiku）
> 原因：文案质量对 Phase 3 出图效果影响大，Sonnet 的创意写作能力明显优于 Haiku
> 成本：20条文案约消耗 2000 token，成本 < ¥0.05/次，可接受

## 六、文案评审机制

> [!danger] 这是整个链路的质量闸门，必须实现，不可跳过
> 没有评审闭环 = AI 质量无法迭代 = 数据库越用越差

### 6.1 评审 UI 设计

```
┌─────────────────────────────────────────────────┐
│  📝 文案候选列表（20条）         筛选：[全部 ▼]  │
├─────────────────────────────────────────────────┤
│                                                   │
│  #1  [痛点激发]  ⭐⭐⭐⭐                          │
│  钩子："站了8小时，脚还不疼？"                    │
│  正文："记忆棉中底，专为久站设计"                 │
│  CTA ："评论问尺码"                               │
│  人群：上班族                                     │
│                                                   │
│  [✅ 采用]  [✏️ 修改后采用]  [❌ 淘汰]           │
│                                                   │
├─────────────────────────────────────────────────┤
│  #2  [社交证明]  ⭐⭐⭐                            │
│  钩子："出门被问了3次在哪买的"                    │
│  正文："独家设计款，撞款率0%"                     │
│  CTA ："点购物车"                                 │
│                                                   │
│  [✅ 采用]  [✏️ 修改后采用]  [❌ 淘汰]           │
│                                                   │
├── ... ──────────────────────────────────────────┤
│                                                   │
│  已选：3条    本次采用率：15%（3/20）             │
│  [确认选定，生成出图 Brief →]                    │
└─────────────────────────────────────────────────┘
```

### 6.2 评审操作定义

| 操作 | 含义 | 后续动作 |
|------|------|---------|
| ✅ 采用 | 直接进入出图流程 | 写入 `approved_copies` 表，采用计数+1 |
| ✏️ 修改后采用 | 策划手动编辑后进入出图 | 保存原版+修改版，两者都写回数据库 |
| ❌ 淘汰 | 不进入出图流程 | 记录淘汰，该公式类型负向权重+1 |

### 6.3 质量监控指标

系统自动统计以下指标，用于 Prompt 版本迭代决策：

| 指标 | 计算方式 | 健康值 |
|------|---------|--------|
| 文案采用率 | 采用数 / 生成总数 | > 15%（20条中至少3条） |
| 公式类型分布 | 各公式类型被采用次数占比 | 无单一类型 > 60% |
| 修改率 | 修改后采用 / 总采用 | < 40%（过高说明生成质量差） |
| 淘汰原因 TOP3 | 人工标注淘汰原因（可选） | 用于定向优化 Prompt |

## 七、Prompt 版本管理

> [!tip] 为什么需要版本管理
> Claude Prompt 改一行可能让采用率从 20% 跌到 5%。版本管理让优化可追溯、可回滚。

### 7.1 版本存储结构

```json
{
  "prompt_version_id": "v1.2",
  "created_at": "2026-03-29",
  "system_prompt": "...",
  "change_note": "新增大促紧迫型公式，调整CTA具体化要求",
  "stats": {
    "total_generated": 200,
    "total_adopted": 38,
    "adoption_rate": 0.19,
    "avg_modification_rate": 0.32
  },
  "status": "active | archived"
}
```

### 7.2 版本切换流程

```
新建 Prompt 版本（v1.3）
    ↓
A/B 测试：连续5次生成，v1.2 和 v1.3 各跑一半
    ↓
对比采用率：v1.3 采用率 > v1.2 × 1.1（提升10%以上）？
    ├── 是 → 切换为 active，v1.2 归档
    └── 否 → 继续用 v1.2，v1.3 标记为 failed
```

## 十一、开发里程碑

| 里程碑 | 内容 | 验收标准 |
|--------|------|---------|
| M1（Week 2）| 参数表单 + Claude API 调用 | 填写参数后能生成20条 JSON 文案 |

| M3（Week 2）| 冷启动判断 + 模板兜底 | 数据库 < 50条时自动切换模板模式，顶部有提示 |
| M4（Week 3）| 数据库检索集成 | ≥ 50条时从数据库检索高频公式注入 Prompt |
| M5（Week 3）| Brief 自动生成 | 采用文案后自动生成 ImageGenerationBrief JSON |
| M6（Week 3）| Prompt 版本管理 | 支持版本记录、采用率统计、版本切换 |
