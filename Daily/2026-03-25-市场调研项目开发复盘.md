# 市场调研项目开发复盘

## 项目全景

本项目分三个独立层级，从轻到重依次构建：

```
Layer 1：市场调研 Skill        ← 纯 Claude 提示词工程，零代码
Layer 2：竞品分析爬虫套件      ← Python + Playwright，本地数据采集
Layer 3：选品工具（产品）      ← PRD + UI + Vue3/NestJS 全栈工程
```

---

## 一、市场调研 Skill（Layer 1）

### 核心成果

一个纯提示词 Skill，触发后强制 Claude 输出五段式结构化报告：品类概览 → 产品清单（≥10款）→ 竞品对比 → 市场趋势 → 综合建议。

### A/B 测试结论

| 维度 | 无 Skill | 有 Skill |
|------|:---:|:---:|
| 信息完整度 | 3/5 | 5/5 |
| 结构清晰度 | 2/5 | 5/5 |
| 数据可用性 | 2/5 | 5/5 |
| 来源可追溯 | 0/5 | 5/5 |
| **总分** | **10/30** | **30/30** |

> **核心结论：认知框架的有无，决定了 AI 输出能否被直接用作交付物。**

### 遇到的问题与解决

**问题 1：Claude 输出结构不稳定**
- 症状：同样的问题，有时给表格，有时给散文，有时跳过某几节
- 解决：在 Skill 里用 Step 编号 + 明确字段约束（"不少于10款"、"必须含发布年份"）固化输出框架

**问题 2：报价数据时效性差**
- 症状：Claude 的训练数据可能滞后 1-2 年，给出过期价格
- 解决：SKILL.md 强制要求每个报价注明来源平台，并加"数据说明"免责块

**问题 3：软件/服务类产品套用硬件框架失效**
- 症状：用同一模板调研 SaaS 产品时，"国际报价（$）"和"核心规格"字段语义错误
- 解决：SKILL.md 加注"软件/服务类：报价改为订阅制/买断/免费，规格改为核心功能点"

### 潜在隐患

- **WebSearch 结果被广告污染**：价格数据可能来自营销页面而非真实售价
- **中小品类信息稀疏**：冷门品类搜不到10款，Claude 会强行凑数
- **汇率未处理**：国内外报价并排，未做当期汇率换算，横向比较有偏差

## 二、竞品分析爬虫套件（Layer 2）

### 技术栈

```
Amazon   → PA-API 5.0（官方 API，无需浏览器）
1688     → Playwright + Cookie 登录
Temu     → Playwright + playwright-stealth（无需登录）
Shopee   → Playwright + playwright-stealth（无需登录）
输出     → Obsidian .md 笔记 / 飞书多维表格（二选一或同时）
```

### 遇到的错误与解决

**错误 1：1688 登录 Cookie 保存路径问题**
- 症状：`cookies/1688_cookies.json` 在 Windows 路径下中文目录名解析失败
- 解决：统一用 `Path` 对象代替字符串拼接，避免路径编码问题

**错误 2：Shopee 地区选择弹窗拦截爬虫**
- 症状：Playwright 打开 Shopee 首先弹出国家/地区选择框，导致后续选择器全部失效
- 解决：在 `ShopeeScraper` 里加入弹窗检测 + 自动关闭逻辑，截图 `debug_1688.png` 辅助调试

**错误 3：Temu 动态渲染商品列表**
- 症状：Playwright 截取 DOM 时商品列表为空，因为懒加载未触发
- 解决：注入 `page.evaluate("window.scrollBy(0, 1000)")` 模拟滚动触发渲染

**错误 4：飞书多维表格写入报 403**
- 最常见原因：应用未被添加为多维表格的协作者（不是权限问题，是"协作者"概念）
- 解决：在文档中单独标注此坑，用 `[!warning]` callout 强调

**错误 5：Amazon PA-API 废弃预警**
- PA-API 5.0 将于 **2026-04-30 废弃**，届时 `scrapers/amazon.py` 需迁移到 Amazon Creators API
- 已在使用说明中注明，模块化设计使迁移只涉及单文件

### 潜在隐患

> [!warning] 三大核心隐患

**1. 反爬封禁风险（高）**
Temu / Shopee 的 Playwright-stealth 方案属于绕过反爬机制，平台有权封 IP 或识别 headless 特征。随时可能失效，且无预警。

**2. Cookie 过期无提示（中）**
1688 Cookie 有效期约 30 天，过期后静默返回未登录页面数据，爬取结果看似正常但字段全为空。未实现自动检测 Cookie 有效性的机制。

**3. 数据结构随平台改版失效（高）**
所有爬虫依赖 CSS 选择器 / XPath，平台一次前端改版就会导致全部字段返回空。无版本锁定机制。

### 技术决策过程中的关键转折

**转折 1：UI 风格从暗紫改为浅色 Teal**
- 初版：Gemini 风格暗紫色（2026-03-17 之前）
- 终版：浅色 Teal（`#0891b2`），参考 Figma AI，白色卡片 + 64px 图标侧边栏
- 原因：暗色方案在跨境电商业务场景下显得过于"极客"，浅色更接近 SaaS 工具感

**转折 2：Figma MCP 能力误判**
- 最初设想：通过 Figma MCP 让 Claude 直接生成设计稿
- 实际情况：Figma MCP 只读，不能写入；生成工具是 Figma Make（独立产品）
- 解决：改用 Gemini Designer Skill 生成 HTML 设计稿，Claude 负责转换和调整

**转折 3：数据采集技术选型**
- 放弃纯爬虫方案
- 最终决策：Reddit snoowrap + Google Trends npm + Apify（Facebook/TikTok）+ 卖家精灵 MCP

### 遇到的问题

**问题 1：NestJS 模块循环依赖**
- `analysis.module` 依赖 `products.service`，`products.module` 又依赖 `analysis.service`
- 解决：用 `forwardRef()` 打破循环，或将共享逻辑抽到独立 `shared.module`

**问题 2：Prisma 在 Windows 路径下生成客户端失败**
- 症状：`prisma generate` 在含中文路径时报错
- 解决：将项目移到纯英文路径，或用 `--schema` 指定绝对路径

**问题 3：Vue3 + Vite 热更新在 WSL2 下失效**
- 症状：文件修改后浏览器不刷新
- 解决：`vite.config.ts` 加 `server: { watch: { usePolling: true } }`

### 潜在隐患

**1. 凭证缺口（阻塞性）**
以下凭证尚未获取，功能无法联调：
- Reddit API Client ID/Secret
- Apify Token
- 飞书 Webhook URL
- Amazon PA-API（2026-04-30 前必须迁移）

**2. 前后端接口未联调**
骨架代码已生成，但 API 合约（字段名、状态码）未经真实数据验证，字段对不上的概率高。

**3. UI 设计稿与实际组件脱节**
HTML 设计稿用硬编码数据，Vue 组件用 Pinia Store 动态数据，视觉效果可能存在差异。

## 四、跨项目共性问题

| 问题 | 出现频率 | 本质原因 |
|------|---------|---------|
| Cookie / Token 过期无感知 | 每个涉及登录的模块 | 缺乏统一的认证状态检测层 |
| 平台接口随时失效 | 爬虫模块 | 依赖非官方接口，无 SLA 保障 |
| Windows 路径中文编码 | 多处 | Python / Node.js 对 Windows 路径的默认处理不一致 |
| 凭证散落多处 | 整个项目 | 缺乏统一的密钥管理（建议统一到 `.env` + Vault） |
