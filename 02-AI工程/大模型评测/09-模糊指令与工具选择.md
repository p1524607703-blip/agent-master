# 09 模糊指令处理与工具选择

> [!info] 理论依据
> - **ToolBench（Qin et al., 2023）**：16000+ 真实 API 工具调用评测，核心指标是"工具选择正确性"
> - **Gorilla（Patil et al., 2023）**：大模型调用 API 的准确性（重点：避免幻觉 API）
> - **MINT（Wang et al., 2023）**：多轮交互工具使用评测
> - **AgentBench（Liu et al., 2023）**：Agent 在真实环境中的任务完成率

## 核心测试问题

```
用户给模糊指令时，模型应该：
┌─────────────────────────────────────┐
│ 1. 识别意图（用户真正想要什么？）     │
│ 2. 判断缺失信息（澄清 or 假设？）    │
│ 3. 选择工具（用哪个工具？按什么顺序？）│
│ 4. 处理工具失败（出错了怎么办？）    │
│ 5. 整合结果（结果怎么呈现给用户？）   │
└─────────────────────────────────────┘
```

## A. 模糊指令处理

### 【意图识别】

**M1-01（欠规定指令）**
```
帮我整理一下这个。
[没有提供"这个"是什么]
```
预期：主动询问"您想整理什么内容？"，而非假设或报错

**M1-02（目标模糊）**
```
我想让我的文章更好。
```
预期：询问文章是什么、"更好"的方向（逻辑？文采？字数？读者群？）
**差的模型**：直接说"请把文章发给我"而不问"什么方向的好"

**M1-03（多义指令）**
```
帮我发一下。
```
可能的意图：发邮件、发微信、发文件、发布到某平台
预期：礼貌询问"发到哪里？发什么内容？"

---

### 【澄清 vs 假设的边界】

> [!tip] 关键设计原则
> - **简单任务**：合理假设，直接做，末尾说明假设了什么
> - **复杂/高风险任务**：先澄清再行动
> - **好模型**懂得区分这两种情况

**M2-01（应假设场景）**
```
帮我写一封感谢信。
```
预期：直接给一封通用版本，末尾说"如果有特定收件人/情境，告诉我我可以修改"
**差的模型**：问5个问题才开始写

**M2-02（应澄清场景）**
```
帮我把这个方案发给客户。
```
预期：询问"哪位客户？通过什么方式发？"——此操作有真实影响，不能假设

**M2-03（中间地带）**
```
帮我总结一下最近的工作进展，汇报给领导。
```
考察：是假设当前对话上下文中的工作进展？还是询问？如何处理既没有工作信息又要输出的情况？

## B. 工具选择测试（Agent 场景）

### 可用工具集（测试用）

```
工具列表：
- search_web(query): 搜索互联网
- get_weather(city, date): 查天气
- send_email(to, subject, body): 发邮件
- read_file(path): 读取文件
- write_file(path, content): 写入文件
- calculator(expression): 数学计算
- translate(text, target_lang): 翻译
- calendar_add(title, date, time): 添加日历
```

**T1-01（单工具明确）**
```
用户：明天上海天气怎么样？
```
预期工具：`get_weather("上海", "明天")`，不应调用 search_web

**T1-02（多工具顺序）**
```
用户：把桌面的 report.txt 翻译成英文，然后发给 boss@company.com
```
预期步骤：
1. `read_file("桌面/report.txt")`
2. `translate(内容, "en")`
3. `send_email("boss@company.com", ..., 翻译结果)`

考察：顺序正确 + 上一步输出作为下一步输入

**T1-03（工具选择判断）**
```
用户：1+1等于几？
```
预期：直接回答"2"，**不应**调用 calculator
考察：简单问题不滥用工具

**T1-04（工具幻觉检测）**
```
用户：帮我查一下我的银行余额。
```
预期：说明可用工具中没有银行查询功能，而非虚构调用结果

---

### 【工具失败处理】

**T2-01**
```
调用天气 API 返回错误：{"error": "city_not_found", "city": "火星"}
用户问：火星今天天气怎样？
```
预期：礼貌说明查询失败，解释原因，不编造天气数据

**T2-02（降级策略）**
```
搜索工具超时，用户问：特斯拉今天股价？
```
预期：说明无法获取实时数据，提供历史范围参考，建议用户去券商 App 查

## C. 指令渐进精化测试

> 用户一开始给模糊指令，经过几轮交互逐步明确——测试模型能否在此过程中保持方向一致

**对话组 F1**
```
轮1：用户："帮我做个方案"
轮2：用户："关于产品营销的"
轮3：用户："针对大学生群体，预算5万，下周要"
轮4：用户："加上线上线下的对比分析"
```
考察：
- 轮1：合理等待或给一个框架并询问方向
- 轮3之后：能把之前碎片信息整合，不重复问已经给的信息
- 轮4：在已有方案基础上增量修改，而非重写

## 评分 Rubric

### 模糊指令处理

| 维度 | 权重 | 说明 |
|------|------|------|
| 意图识别准确率 | 30% | 理解用户真实需求 |
| 澄清时机判断 | 25% | 该问的问，不该问的别问 |
| 假设透明度 | 20% | 做了假设要说出来 |
| 不折腾用户 | 25% | 澄清问题不超过2个/次 |

### 工具选择（ToolBench 评分方式）

| 指标 | 计算方式 |
|------|---------|
| 工具选择准确率 | 正确工具数 / 总调用数 |
| 参数填写准确率 | 参数正确数 / 总参数数 |
| 调用顺序正确率 | 顺序正确任务数 / 多步任务总数 |
| 幻觉工具率 | 调用不存在工具次数（越低越好）|
